# 执行评估准则

当 TODO 文档含糊，或你需要比 `SKILL.md` 顶层工作流更严格的质量标准时，使用此准则。

## 保持顺序

- 将文档顺序作为默认优先级。
- 除非文档明确说明相反含义，否则将章节标题视为阶段。
- 当前任务仍可推进时，不要跳到后面的任务。
- 如果当前任务被外部事项或未解决问题阻塞，记录阻塞原因，并且只继续执行不依赖该阻塞工作的后续任务。

## 严格定义“完成”

只有当任务文本直接或间接要求的所有交付物都完成时，任务才算完成。

示例：

- 代码变更任务：实现行为；如果行为变化，更新邻近测试；运行最相关的验证。
- 重构任务：保持行为不变；按需更新调用方；运行仍能覆盖受影响区域的最小测试。
- 文档任务：更新任务实际指向的源文档，而不是只写注释或临时笔记。
- 调研任务：产出被要求的结论或产物，而不仅是原始探索记录。
- 复选框同步：真实工作完成后才标记 Markdown 复选框；如果文件使用普通列表项而不是复选框，应单独汇报完成情况，不要自行重写文档结构。

## 避免作弊

- 不要把分析、脚手架或部分接线称为“完成”。
- 不要留下 `TODO`、`FIXME`、占位符或无效桩函数来替代被要求的工作，除非任务明确要求搭建脚手架。
- 不要通过重写 TODO 文件来隐藏未完成工作。
- 不要因为后面的简单低价值条目更容易，就优先处理它们；除非前面的困难条目确实被阻塞。
- 不要声称运行了你实际没有运行的验证。

## 最大化有用进展

- 完成一个条目后，积极复用上下文，直接进入下一项。
- 如果相邻两个任务触及相同文件，趁上下文仍热时继续推进，但汇报时仍作为独立完成项分别说明。
- 如果某个任务暴露出后续条目需要的小型前置工作，立即完成它，不要留下隐藏陷阱。
- 当任务被阻塞时，只在文档顺序允许且诚实的范围内继续前进。

## 精确描述阻塞

用具体术语记录阻塞：

- 缺少凭证、令牌、硬件或网络访问。
- 与本次改动无关的基线失败。
- 仓库指令互相冲突。
- 存在会产生不同用户可见结果的重大解释分歧。

避免使用“需要更多工作”或“很复杂”等含糊阻塞描述。
