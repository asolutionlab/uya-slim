# Uya 微容器虚拟化系统需求文档

**版本**: v1.3  
**日期**: 2026-04-02  
**目标平台**: CH32V003（主目标）+ ESP32 + RV1106  
**载体语言**: Uya v0.74  
**状态**: 评审修复版

---

## 1. 目标与范围

本需求文档定义在 512KB 物理内存、无硬件 MMU、无硬件虚拟化支持前提下的微容器系统。设计目标如下：

- 崩溃不扩散：单容器异常不影响内核和其他容器。
- 更新不中断：支持热更新与失败回滚。
- 行为可预测：所有核心路径具备可计算上界。
- 安全可审计：安全决策在编译期、链接期、加载期、运行期形成闭环。

本版本聚焦规格闭环，不包含完整实现代码。

### 1.1 与 capability runtime 的关系

本文档中的“微容器”是执行与隔离层，不是能力包协议本身。

在当前仓库里，两者建议按如下方式配合：

- `capability runtime` 负责定义能力包 manifest、权限模型、Host API、生命周期与后端兼容层
- “微容器系统”负责提供容器镜像校验、地址空间隔离、系统调用边界、调度、崩溃恢复与更新回滚

因此：

- `capability` 是“要运行什么”的抽象
- “微容器”是“如何安全运行它”的机制

后续若以微容器承载 capability，则 capability manager 负责安装和状态管理，微容器 runtime 负责真正装载镜像、建立隔离上下文并执行。

## 2. 设计原则

### 2.1 确定性优先

- 禁止容器态动态分配器（容器不允许直接 `malloc`）。
- 禁止递归调用和不可界定循环。
- 调度、地址翻译、系统调用验证路径必须给出最坏时延上界。

### 2.2 零隐式行为

- 不存在隐式硬件 MMU 调用。
- 所有容器态访存检查由编译器显式注入。
- 所有外部交互必须经系统调用进入内核代理。

### 2.3 故障隔离

- 容器崩溃后仅回收该容器资源，不触发全局重启。
- 通过页表权限位和调度隔离阻断容器间越权访问。
- 硬件看门狗仅作为系统级兜底，不替代容器级恢复。

### 2.4 可审计性

- 权限能力由编译参数控制，不由隐式上下文推断。
- 镜像加载前必须完成哈希、重定位、指令策略三重验证。

## 3. 修复后的内存架构（512KB 闭合）

### 3.1 物理地址空间总览

物理内存总量为 512KB，按固定分区管理。以下预算总和严格等于 512KB：

- 中断向量与启动代码：1KB
- 内核代码段（调度、页表、syscall、驱动核心）：20KB
- 内核数据段（CCB、全局队列、页池元数据）：8KB
- 容器代码区（8 槽位，每槽 24KB）：192KB
- 容器数据区（8 槽位，每槽 20KB）：160KB
- 共享内存区（IPC 缓冲、DMA 共享）：64KB
- 升级暂存区（热更新 staging）：32KB
- 中断栈：4KB
- 内核主栈：8KB
- 对齐与守卫保留：7KB

### 3.2 分区约束

- 容器代码区只读执行，容器数据区读写不可执行。
- 共享内存区按能力令牌分段授权，不允许默认全可见。
- 升级暂存区与活跃代码区必须物理隔离，防止覆盖式升级。

### 3.3 分配策略

- 内核采用固定页池分配，不使用通用堆分配器。
- `SYS_ALLOC` 仅在容器数据区配额内进行页映射，失败返回配额错误。

## 4. 权限模型与编译模式

### 4.1 编译参数

编译时必须显式指定：

- `--app app`
- `--app microapp`

同一源码可在两种模式下生成不同安全级别产物。

### 4.2 app 模式能力

- 允许 `@asm`
- 允许 `extern`
- 允许物理地址字面量解引用
- 生成直接内存访问代码

### 4.3 microapp 模式限制

- 禁止 `@asm`，违规则报 `E4001`
- 禁止 `extern`，违规则报 `E4002`
- 禁止物理地址字面量访问设备寄存器区，违规则报 `E4003`
- 指针解引用自动注入 `uya_mmu_translate`
- 数组访问自动注入边界检查

---

## 5. 统一页表方案（单级固定页）

### 5.1 选型结论

本版本统一采用单级固定页方案，不再使用“2MB 区段 + 可变粒度”混合表述。

### 5.2 地址空间定义

- 每容器虚拟空间：8MB
- 页大小：64KB
- 页表项数量：128
- 每项映射一个 64KB 虚拟页

### 5.3 PTE 布局（32 位）

- bit31: `V`（有效）
- bit30: `R`（可读）
- bit29: `W`（可写）
- bit28: `X`（可执行）
- bit27: `U`（用户可访问）
- bit26: `S`（共享页）
- bit25: `D`（脏）
- bit24: `A`（访问过）
- bit23..7: `PPN`（17 位物理页号）
- bit6..0: 保留

### 5.4 翻译公式

- `vpn = (vaddr >> 16) & 0x7F`
- `offset = vaddr & 0xFFFF`
- `paddr = (pte.ppn << 16) | offset`

权限错误、无效页、越界均转为容器故障码并触发容器级异常处理。

### 5.5 删除软件 TLB 的依据

- 单级表查找路径固定，时延可界定。
- 软件 TLB 维护会引入替换策略分支和额外状态同步，增加不可预测性。
- 在 48MHz 和 128 项规模下，直接查表可满足本项目时延目标。

## 6. 系统调用接口（总览）

容器与内核唯一交互通道为 `syscall(number, arg1, arg2)`。具体 ABI 细节见：

- `docs/microcontainer/syscall_abi.md`

核心调用包括：

- `SYS_PRINT`
- `SYS_ALLOC`
- `SYS_IO`
- `SYS_YIELD`

若上层承载对象为 capability，则这里的 syscall 不直接暴露给产品层，而是通常经由统一 Host API 间接访问。也就是说：

- capability 侧看到的是 `host_time_now`、`host_kv_get`、`host_vibrate` 这类能力接口
- 微容器侧负责把这些接口翻译为受控 syscall 或内核代理调用

## 7. 容器生命周期与调度

### 7.1 容器状态

- `EMPTY`
- `LOADING`
- `READY`
- `RUNNING`
- `BLOCKED`
- `CRASHED`
- `ZOMBIE`

若容器承载的是 capability，则建议建立如下近似映射：

- capability `Installed` -> 容器 `LOADING` / `READY`
- capability `Active` -> 容器 `RUNNING` / 可调度
- capability `Disabled` -> 容器停止调度或实例已回收
- capability `Failed` -> 容器 `CRASHED` 或被策略熔断

这样可以避免产品层状态机与底层容器状态机各说各话。

### 7.2 调度策略

- 256 优先级，按 8 组管理，每组 32 级。
- 使用位图选择最高优先级非空组，组内轮转。
- 时间片为 1ms。
- `yield()` 为显式协作点。

### 7.3 崩溃恢复

- 崩溃现场写入持久日志。
- 重启次数未超限时执行容器级重建。
- 超限则熔断为 `CRASHED`，等待外部干预或策略回收。
- 热更新原子切换与崩溃恢复细则见 `docs/microcontainer/update_recovery.md`。

## 8. 加载期验证（总览）

加载容器镜像必须依次通过：

- SHA-256 完整性验证
- 重定位与符号约束验证
- 指令策略验证（禁用越权指令模式）

详细规则见：

- `docs/microcontainer/image_validation.md`

## 9. C99 后端要求

### 9.1 运行时接口

需要提供：

- 页表结构定义
- `uya_mmu_translate` 与故障返回语义
- 内存屏障宏和调用约束

### 9.2 模式化生成策略

- `kernel` 模式生成直接访存
- `microapp` 模式生成翻译包装与权限检查；对外可理解为在 app 基础上启用更严格约束

### 9.3 编译选项约束

- `-O2`
- `-fno-strict-aliasing`
- 对关键屏障函数禁内联与跨函数重排

---

## 10. 安全边界

### 10.1 编译期

- 模式约束和禁用特性在 AST/Checker 阶段静态拒绝。

### 10.2 链接期

- 容器目标文件强制容器标签。
- 拒绝未授权外部符号与内核私有符号引用。

### 10.3 加载期

- 镜像元数据、签名哈希、重定位、指令策略联合校验。

### 10.4 运行期

- 每次容器态敏感访存经页表权限检查。
- 关键内核路径使用中断保护与屏障保证一致性。

## 11. 性能目标（可验收口径）

性能目标与测试方法见：

- `docs/microcontainer/benchmark_plan.md`

本需求文档仅保留验收结论阈值：

- 空循环开销：不超过 1.02x
- 指针解引用开销：冷路径不超过 18x，热路径不超过 8x
- 数组顺序遍历开销：不超过 1.6x
- 上下文切换开销：不超过 12x
- syscall 往返开销：不超过 12x

## 13. 风险与缓解

### 13.1 软件翻译开销超标

- 通过编译期证明消除可证明安全的冗余检查。
- 热路径采用批量翻译和缓存化短窗口策略。

### 13.2 编译器优化破坏屏障语义

- 使用 `volatile` 访问点、内存栅栏宏、受限优化参数。

### 13.3 页表遍历重入

- 关键段禁中断，采用短临界区，避免嵌套陷入。

### 13.4 热更新中断失败

- 双槽版本标记、原子切换、失败回滚、看门狗兜底。

### 13.5 容器逃逸

- 编译期禁 `@asm/extern/物理地址`
- 链接期禁非法符号
- 加载期禁越权指令模式
- 运行期页表权限二次约束

---

## 14. 结论

本 v1.3 版本完成了评审阻断项修复，形成了可实现规格：

- 内存预算闭合到 512KB
- 页表模型从冲突叙述收敛为单级固定页
- syscall、加载验证、性能验收有明确落地文档
- 热更新与崩溃恢复具备原子切换与回滚路径

系统下一步进入实现阶段，可按“编译器 -> 后端运行时 -> 内核最小可用 -> 验证压测”的顺序推进。

---

## 15. ESP32 与 RV1106 实现方案

本需求在 CH32V003 上定义最小确定性基线，同时补充 ESP32 与 RV1106 的迁移方案。平台细则见：

- `docs/microcontainer/platform_impl.md`

实现约束如下：

- CH32V003 仍作为规范基线，不因新平台引入而放宽安全规则。
- ESP32 按无 MMU 兼容路径实现，优先保证行为一致性。
- RV1106 支持 `baremetal_compat` 与 `linux_hosted` 双路径，但容器安全语义保持一致。
