# 自举编译器内存优化计划

## 问题分析


| 组件                | C 编译器 | 自举编译器  | 倍数  |
| ----------------- | ----- | ------ | --- |
| arena_buffer      | 32MB  | 384MB  | 12x |
| temp_arena_buffer | 32MB  | 192MB  | 6x  |
| 总计                | ~64MB | ~576MB | ~9x |


根因主要有三：

1. **temp_arena 累积所有依赖文件的 AST**：`collect_module_dependencies` 递归解析每个文件，AST 累积不释放，峰值等于所有依赖文件 AST 之和
2. **Uya 结构体比 C 大**：ASTNode 等从 C 的 union 变为 flat struct，节点体积约为 C 的 2–3 倍
3. **分配顺序差异**：自举生成的 C 与手写 C 的分配顺序略异，导致峰值更高

## 优化策略

### 策略 1：temp_arena 在依赖收集时按文件回滚（核心优化，预期明显降内存）

依赖收集时，每个文件解析后只需 `extract_use_modules` 提取模块名，完整 AST 可以立即丢弃。  
通过 `arena_save/arena_restore` 在每次 `extract_use_modules` 后回滚，可将 temp_arena 峰值从「所有依赖 AST 之和」降为「单文件 AST + 路径字符串」。

**实现步骤：**

1. 在 [arena.uya](src/arena.uya) 添加：
  - `arena_save(arena: &Arena) usize`：返回 `arena.offset`
  - `arena_restore(arena: &Arena, offset: usize) void`：设置 `arena.offset = offset`
2. 在 [main.uya](src/main.uya) 的 `collect_module_dependencies` 中，在 `extract_use_modules` 之后、递归之前插入回滚逻辑：

```uya
// 第 571 行 parser_parse 之后，第 579 行 extract_use_modules 之后
const save_offset: usize = arena.offset;  // 在 parser_parse 之前保存（此时只有祖先分配的 path_copy）
// ... parser_parse ...
// ... extract_use_modules ...
arena.offset = save_offset;  // 回滚，释放当前文件的 AST（path_copy 由调用方在更早 offset 分配）
```

注意：`path_copy` 的分配时机需核对。父调用在递归前分配 `path_copy`，因此 `save_offset` 应在进入 `collect_module_dependencies` 时、在 `parser_parse` 之前记录。更稳妥的顺序是：

- 函数入口处：`save_offset = arena.offset`
- 解析当前文件、提取模块
- `arena.offset = save_offset`（释放 AST，保留更早的 path_copy）
- 对每个依赖模块分配 path_copy 并递归

这样递归时每个子调用只占用「当前文件 AST」的大小，不再累积。

1. C 编译器侧：在 [compiler-c/src/arena.c](compiler-c/src/arena.c) / [arena.h](compiler-c/src/arena.h) 同步增加 `arena_save`、`arena_restore`，并在 [compiler-c/src/main.c](compiler-c/src/main.c) 的 `collect_module_dependencies` 中加入相同回滚逻辑，保证自举对比 (make b) 通过。

**预期效果**：temp_arena_buffer 可从 192MB 降到约 16–32MB。

---

### 策略 2：实测峰值并缩减 arena_buffer / temp_arena_buffer

当前 384MB/192MB 多为经验值（「保证 make b 通过」），可能偏保守。

1. **增加可选调试输出**：
  在 [arena.uya](src/arena.uya) 的 `arena_alloc` 中，当 `UYA_ARENA_DEBUG` 环境变量存在时，定期（如每 1MB 分配）向 stderr 输出 `arena.offset`。
2. **分阶段测量**：
  在 main 中若干关键点输出 offset：
  - 依赖收集结束
  - 所有文件解析 + 合并完成
  - checker 结束
  - codegen 结束
   便于定位峰值出现阶段。
3. **二分查找最小可用缓冲**：
  在完成策略 1 后，逐步减小 `ARENA_BUFFER_SIZE` 和 `temp_arena_buffer`，每次运行 `./compile.sh --c99 -e -b`，找到最小仍能通过的值。  
   **建议目标**（在策略 1 生效后）：  
  - temp_arena_buffer: 32MB（与 C 一致或略大）  
  - arena_buffer: 64–96MB（考虑 Uya 结构体更大，留一定裕量）

---

### 策略 3：对齐 C 与自举的 arena 使用顺序（可选，降低 arena_buffer 需求）

若策略 1+2 后 arena_buffer 仍明显大于 C，可排查自举生成 C 与手写 C 的调用顺序差异。

- 对比 `compiler-c` 与 `src` 生成的中间 C：  
重点关注 parser、checker、codegen 中对 `arena_alloc` 的调用顺序，以及何时分配 AST / Type / Symbol 等大对象。  
- 在自举 Uya 源码中调整调用顺序或拆分函数，使 arena 使用模式更接近 C 版本，从而降低峰值。

---

### 策略 4：Arena 接口与兼容性

`arena_save` / `arena_restore` 需在 C 与 Uya 两套实现中保持一致，以保证：

- 自举生成的 C 与手写 C 行为一致  
- `make b` 自举对比通过  
- 测试 `test_arena_buffer` 如涉及 arena 行为，需相应更新

---

## 实现顺序建议

1. **阶段 1（高收益、低风险）**
  - 实现 arena_save/arena_restore（arena.uya + arena.c/h）  
  - 在 `collect_module_dependencies` 中引入回滚逻辑  
  - 将 temp_arena_buffer 降至 32MB，验证 `make b` 通过
2. **阶段 2（验证与调优）**
  - 添加 UYA_ARENA_DEBUG 输出，测量各阶段 arena 峰值  
  - 二分缩小 arena_buffer（如从 384MB 逐步降到 128MB、96MB、64MB），每次跑 `compile.sh --c99 -e -b`
3. **阶段 3（可选）**
  - 若 arena_buffer 仍偏高，按策略 3 分析并调整分配顺序

## 风险与注意点

- **path_copy 生命周期**：回滚时绝不能覆盖仍被 `file_list` 引用的 path_copy。需确认 path_copy 均在 `save_offset` 之前分配。  
- **自举循环**：修改 `arena.uya` 后，需先用当前自举编译器成功编译，再用于自举；若 arena 行为变化导致无法完成自举，需保留足够大的缓冲作为回退。  
- **测试**：每次修改后必须跑 `./tests/run_programs.sh --uya --c99` 和 `./compile.sh --c99 -e -b`，确保功能与自举对比均通过。
