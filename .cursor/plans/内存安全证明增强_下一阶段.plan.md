# 内存安全证明增强计划（下一阶段）

## 概述

v0.5.0 已实现基础的内存安全证明系统，包括常量折叠、路径敏感分析、符号执行引擎、证明超时机制等。本计划旨在进一步增强系统能力，扩展类型支持、增强表达式分析、支持跨函数分析，以及 IDE 集成。

## 当前状态

### v0.5.0 已完成 ✅

- ✅ 常量折叠增强（溢出/越界/除零检测）
- ✅ 路径敏感分析框架（约束收集和传播）
- ✅ 符号执行引擎（线性约束求解）
- ✅ 证明超时机制（步数限制）
- ✅ 未初始化使用检测
- ✅ 空指针解引用检测
- ✅ 证明失败错误报告（带修复建议）
- ✅ 391 个测试全部通过

### 待改进 ❌

- ❌ `usize` 类型不支持约束系统
- ❌ 非线性表达式（乘法、除法）分析能力有限
- ❌ 仅支持函数内分析，无跨函数传播
- ❌ 无 IDE 集成支持

## 实现目标

### 1. 支持 `usize` 类型约束

**目标**：扩展约束系统支持 `usize` 无符号整数类型

**背景问题**：
- 当前约束系统仅支持有符号整数类型（`i32`, `i64` 等）
- `usize` 类型变量无法参与约束计算
- 标准库中多处使用 `usize`，需要临时改为 `i32` 来绕过限制

**实现内容**：

- 扩展 `ConstraintValue` 支持 `usize` 类型
- 修改约束收集逻辑，正确处理无符号比较：
  - `usize` 变量默认下界为 0
  - `usize` 比较时考虑无符号语义
- 更新约束求解器，支持 `usize` 算术运算
- 添加 `usize` 相关测试用例

**涉及文件**：
- `src/checker.uya` - 扩展约束系统数据结构和算法
- `lib/libc/stdlib.uya` - 恢复原有 `usize` 类型
- `lib/std/io/file.uya` - 恢复原有 `usize` 类型

**测试用例**：
- `test_usize_constraints.uya` - usize 约束测试

**预估工作量**：中等（2-3 天）

---

### 2. 增强非线性表达式分析

**目标**：支持乘法、除法等非线性约束求解

**背景问题**：
- 当前约束系统仅支持线性表达式（加法、减法）
- 数组索引计算中常用的 `i * stride` 无法正确分析
- 导致一些合法的边界检查无法通过证明

**实现内容**：

- 扩展 `SymbolicExpr` 支持非线性运算：
  - 乘法：`a * b`
  - 除法：`a / b`、`a % b`
  - 移位：`a << n`、`a >> n`
- 实现非线性约束求解策略：
  - 符号边界传播（保守估计）
  - 区间算术（Interval Arithmetic）
  - 模式匹配特殊形式（如 `i * 2 < len`）
- 添加非线性表达式测试用例

**涉及文件**：
- `src/checker.uya` - 扩展符号表达式和求解器

**测试用例**：
- `test_nonlinear_bounds.uya` - 非线性边界检查测试

**预估工作量**：较高（5-7 天）

---

### 3. 支持跨函数约束传播

**目标**：通过函数契约实现跨函数安全证明

**背景问题**：
- 当前证明范围仅限函数内
- 无法利用函数调用处的约束信息
- 无法证明涉及函数调用的复杂场景

**实现内容**：

- 设计函数契约语法：
  ```uya
  fn get_item(arr: []i32, idx: usize) -> i32
      requires idx < @len(arr)
  {
      return arr[idx];
  }
  ```
- 实现契约收集：
  - 解析 `requires` 和 `ensures` 子句
  - 存储契约信息到函数签名
- 实现契约验证：
  - 调用处验证 `requires` 条件满足
  - 返回处验证 `ensures` 条件成立
- 支持契约继承和组合

**涉及文件**：
- `src/parser.uya` - 添加契约语法解析
- `src/checker.uya` - 实现契约验证逻辑
- `docs/uya_ai_prompt.md` - 更新语言规范

**测试用例**：
- `test_function_contracts.uya` - 函数契约测试
- `test_cross_function_proof.uya` - 跨函数证明测试

**预估工作量**：高（7-10 天）

---

### 4. IDE 集成错误提示

**目标**：提供友好的 IDE 错误提示和修复建议

**背景问题**：
- 当前错误消息仅通过终端输出
- 无法与 IDE 集成提供实时反馈
- 修复建议不够直观

**实现内容**：

- 设计 LSP（Language Server Protocol）接口：
  - 诊断信息（Diagnostics）
  - 代码补全（Completion）
  - 悬停提示（Hover）
  - 代码操作（Code Actions）
- 实现 LSP 服务器：
  - 解析 LSP 协议
  - 提供诊断和修复建议
  - 支持增量更新
- 生成错误修复代码片段：
  - 数组边界检查代码
  - 空指针检查代码
  - 溢出检查代码

**涉及文件**：
- `src/lsp.uya` - 新增 LSP 服务器实现
- `src/main.uya` - 添加 LSP 模式入口

**测试用例**：
- 手动测试与 VS Code / 其他 IDE 集成

**预估工作量**：高（10-14 天）

## 技术难点

1. **usize 类型处理**：无符号整数的边界条件和溢出行为与有符号不同
2. **非线性约束求解**：非线性约束求解是 NP 困难问题，需要权衡精度和性能
3. **跨函数分析**：需要设计合适的契约语法和验证策略
4. **LSP 协议**：需要完整实现 LSP 协议，与各种 IDE 兼容
